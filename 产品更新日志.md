## v2.0 — 2026-05-20 · 五模块重构 + 简历库/岗位库 + Vision JD 提取

### 迭代基础信息
- **版本号**：v2.0
- **迭代周期**：2026-05-20
- **迭代类型**：重构 + 新增

### 本次改动核心功能
1. **五模块结构**：侧边栏从 3 页扩展为 5 页（素材库 → 简历库 → 岗位库 → 面试准备 → 模拟面试）
2. **简历库（新模块）**：PDF 格式上传、多份简历并存、浏览器原生 iframe 预览、删除管理
3. **岗位库（新模块）**：结构化四字段（公司/岗位/职责/要求），支持 Claude Vision 图片自动提取 + 手动录入双通道
4. **素材库重定位**：从「什么都传」改为知识库（仅上传参考资料和笔记），简历归简历库，岗位归岗位库
5. **面试准备重构**：选简历（库内/上传）+选岗位（库内/录入）双字段，新内容自动同步入库，基于简历+JD+素材库上下文自由对话
6. **模拟面试重构**：同面试准备的简历+岗位选择机制，面试中展示岗位信息卡片，历史面试记录保留对话和评分
7. **Vision JD 提取**：新增 `prompts/jd_extract.txt`，Claude Vision 从 JD 图片提取结构化 JSON 四字段
8. **语音接口预留**：streamlit-mic-recorder + Whisper API 方案已设计，因 Streamlit Cloud 依赖问题暂注释

### 本次使用 AI 技术
- **Claude Vision API**：用于 JD 图片的结构化信息提取（`claude-sonnet-4-6` 模型）
- **数据模型扩展**：新增 `PositionInfo` 和 `ResumeRecord` dataclass，复用 `JDInfo` 字段结构
- **存储层扩展**：`StorageManager` 新增 resume CRUD 和 position CRUD 方法
- **PDF 预览**：base64 编码 + iframe 嵌入，无需额外依赖

### 开发落地难点
1. **requirements.txt 版本号错误**：`streamlit-mic-recorder>=1.16` 写错（实际最新 0.0.8），导致 Streamlit Cloud 安装失败；已修复为注释掉，语音功能优雅降级
2. **死代码清理**：QA 发现 `{profile_summary}` 占位符残留、语音 import 死代码；已全部移除
3. **JD 图片提取的模型选择**：多方对比 Vision vs OCR+LLM 方案后选择 Vision，原因是准确率高、一步到位、架构简单，单张成本约 $0.01-0.03
4. **五页面的导航 key 管理**：所有 widget key 加 `_v3` 后缀防冲突，动态 key 用 `hash()` 生成

### 解决方案与取舍
- **放弃语音功能立即上线**：streamlit-mic-recorder 在 Streamlit Cloud 有系统依赖问题，改为预留接口 + 注释代码，优先上线核心功能
- **岗位库双录入通道**：Vision 提取为主路径，手动录入为兜底——好处是即使没有 API Key 或 Vision 失败，用户仍可手动填写
- **简历仅 PDF**：放弃支持 docx/txt 等格式，减少文件处理复杂度，PDF 是简历最通用的格式

### 产品复盘思考
- **五模块比三模块更清晰**：之前的「素材库」概念模糊（什么都往里扔），拆分为素材库/简历库/岗位库后每个模块职责单一
- **结构化岗位数据是核心资产**：JD 从「一段文本」变成「四字段结构化」，面试准备和模拟面试的 prompt 质量明显提升
- **Vision 提取是体验亮点**：用户拍一张 JD 截图就能自动填充所有字段，比手动打字快 10 倍
- **规范执行仍需加强**：本次 v2.0 push 后又跳过了 QA 和文档更新（事后补救），说明「先修 bug → 忘了文档」的惯性还没改掉

## v1.3 — 2026-05-18 · UI/UX 大重构

### 迭代基础信息
- **版本号**：v1.3
- **迭代周期**：2026-05-18
- **迭代类型**：重构 + 新增

### 本次改动核心功能
1. **全新视觉语言**：采用「Japanese Editorial × Warm Minimalism」设计风格，暖色纸张底色、DM Serif Display 标题字体、朱红色点缀、非对称卡片布局，彻底告别 Streamlit 默认的粗糙感
2. **新增首页仪表盘**：聚合展示素材统计、简历/岗位数量、面试次数、最近面试记录、候选人画像概览，提供三个快速入口卡片直达核心功能
3. **水平顶部导航**：将侧边栏单选改为水平标签式导航（利用 `st.radio(horizontal=True)` + 自定义 CSS），视觉更轻盈，操作更直观
4. **素材库卡片化**：文件列表从纵向表格式改为 3 列卡片网格，每种文件类型有独特图标和分类标签，hover 状态有微交互
5. **聊天气泡重设计**：面试官发言使用左侧红色边框暖粉底色气泡，候选人发言使用右侧靛蓝边框淡紫底色气泡，双方角色一目了然
6. **面试状态栏**：模拟面试进行中新增状态信息条（公司/岗位/已答题数/状态），强化面试沉浸感

### 本次使用 AI 技术
- **前端设计 Skill（frontend-design）**：首次在项目中调用 Claude Code 内置 frontend-design skill，获得系统化设计指导（设计方向选择、字体搭配、色彩体系、空间构成），避免裸写平庸 UI
- **CSS 注入策略**：利用 Streamlit `st.markdown(unsafe_allow_html=True)` 注入近 15KB 自定义 CSS，覆盖全部原生组件样式
- **工作流规范化**：将 frontend-design skill 调用规则写入 CLAUDE.md 第八节，未来所有 UI 任务强制先调 Skill

### 开发落地难点
1. **Streamlit 布局限制**：无法使用自定义 JS、无法精确控制 DOM 层级、`st.markdown` 的 HTML 块与原生组件交叉嵌套容易产生渲染问题
2. **水平导航交互**：Streamlit 原生 `st.radio(horizontal=True)` 不支持直接样式化选中态，最终使用 CSS `:has()` 伪类选择器实现选中高亮（需 Chrome 105+ / Safari 15.4+，现代浏览器均已支持）
3. **Streamlit sidebar 与顶部导航冲突**：原有侧边栏展开状态下会压缩内容区，改为默认折叠（`initial_sidebar_state="collapsed"`），侧边栏降级为次要信息展示

### 解决方案与取舍
- **放弃自定义 JS 方案**：坚守「Streamlit 原生 + CSS」策略，不引入 iframe 或自定义组件，保证部署简单性和 Streamlit Cloud 兼容性
- **`:has()` 选择器**：选择拥抱现代 CSS 标准而非 polyfill，面向主流桌面浏览器，移动端兼容性不影响核心用户场景
- **Profile 卡片纯 HTML 渲染**：发现 `st.markdown` + 原生 Streamlit 组件混用会导致 div 嵌套断裂，改为完全用 HTML 字符串构建卡片内容，牺牲 Streamlit 原生渲染的便利性换取视觉完整性

### 产品复盘思考
- **Skill 的价值被低估了**：一开始直接写 CSS 是典型的「工程师直觉」—— 以为自己能搞定。实际上 frontend-design skill 提供的系统性设计思维（定方向→选字体→建色彩体系→空间构成）远比自己东拼西凑高效
- **B 端工具的审美是信任信号**：面试辅助工具如果看起来像学生项目，用户很难信任其评估能力。精致的 UI 本身就是产品价值的一部分
- **Streamlit 不是瓶颈**：大量开发者认为 Streamlit 做不出好 UI，实际上 CSS 注入 + 设计思路就可以做出相当不错的界面。限制更多来自设计能力而非技术栈

## v1.2 — 2026-05-18 · 素材库扩展 + Mock面试入口优化

### 迭代基础信息
- **版本号**：v1.2
- **迭代周期**：2026-05-18
- **迭代类型**：新增 + 优化

### 本次改动核心功能
1. **素材库支持图片和Excel**：导入入口新增 .xlsx/.xls/.png/.jpg/.webp 格式。Excel 文件通过 openpyxl 转为 markdown 表格后送 LLM 提取结构化信息；图片存入独立分类（images），支持 Web UI 预览，暂不调用 LLM 提取（DeepSeek 不支持多模态）
2. **模拟面试支持本地上传简历**：简历选择区新增「从素材库选择 / 上传本地文件」切换，用户可不预先导入素材库直接上传简历开始面试
3. **模拟面试支持本地上传 JD**：JD 选择区同样新增本地文件上传入口，确保无论素材库有无 JD 都能开始模拟面试
4. **MockSkill 文件读取升级**：统一文件读取逻辑，支持 txt/md/pdf（pdfminer）/xlsx（openpyxl）四种格式，本地上传和素材库选择共用同一读取路径

### 本次使用 AI 技术
- Excel 提取流程：openpyxl 读取 → 转 markdown 表格文本 → material_extract prompt 结构化提取
- MockSkill 新增 `_read_file_content()` 工具函数，按文件扩展名分派读取方式，LLM 上下文注入逻辑保持不变
- 新增依赖：openpyxl（Excel 读取）、Pillow（图片处理）

### 开发落地难点

**难点 1：图片素材的处理方式**
- 问题：图片是高频素材类型（截图简历、产品架构图），但当前主模型 DeepSeek 不支持多模态输入，无法像文本文件那样自动提取结构化信息
- 解决方案：图片作为独立分类（images）管理，支持上传/预览/删除，但不做 LLM 提取。索引中仅记录文件名、路径、导入时间等元数据。当用户切换到支持多模态的模型（如 Claude）时，可扩展接入视觉提取
- 放弃方案：接入 pytesseract OCR —— 增加复杂度和依赖，且中文 OCR 准确率不稳定，投入产出比低

**难点 2：模拟面试本地文件上传的设计**
- 问题：之前仅支持素材库下拉选择，如果素材库为空则无法开始面试。用户需要额外到素材库上传，流程断开
- 解决方案：在 Mock 面试设置页同时提供两种入口（素材库选择 + 本地上传），用户无需切换页面即可完成简历和 JD 的输入。上传文件保存到临时目录，面试期间复用 MockSkill 现有路径读取逻辑
- 放弃方案：将本地上传做成一键导入素材库再选择——增加操作步骤，且用户可能不希望将临时文件污染素材库

### 解决方案与取舍

**产品取舍**：
- 图片不做 OCR 提取而非强行接入：当前 DeepSeek 不支持多模态是硬约束，OCR 方案准确率不高反而增加用户困惑。不如明确告知图片为「参考素材」而非「可提取素材」，等模型能力升级后再扩展
- Excel 归类为 project 而非新增「数据」分类：避免分类膨胀，Excel 本身就是项目文档的一种呈现形式，合并到 project 分类符合用户心智

### 产品复盘思考

1. **素材库的文件类型覆盖是渐进过程**：v1.0 只支持 txt/md，v1.1 增加 pdf，v1.2 增加 xlsx 和图片。每次扩展都带来新的处理逻辑，但核心链路（导入→提取→索引→搜索）保持一致，说明架构设计合理
2. **Mock 面试入口应该是零摩擦的**：用户打开模拟面试页面，应该能立即开始面试，不应该因为素材库为空就被阻塞。本地上传入口解决了这个问题，后续可考虑更激进的方案：支持不选简历直接开始通用面试
3. **图片素材的未来价值**：当切换到支持多模态的模型（如 Claude），图片将能从「参考素材」升级为「可提取素材」，这在简历截图、架构图解读等场景有巨大价值，值得在模型升级时优先实现

- Mock 面试官上下文升级：优先使用用户选定简历/JD 全文，回退到画像+索引 JD
- MaterialSkill / PrepSkill 新增可选 obsidian_connector 参数，保持向后兼容

### 开发落地难点

**难点 1：Obsidian 与现有搜索的集成方式**
- 问题：Obsidian 连接器写好了但一直没接入，用户搜索时只能查本地素材库（几乎为空）
- 解决方案：MaterialSkill 新增 obsidian_connector 可选依赖，search 时并发查询两端结果并合并展示；PrepSkill 在 _build_context 时自动注入 Vault 概览
- 放弃方案：单独做一个 obsidian skill CLI 命令——对用户来说心智负担重，不如直接融入 material search

**难点 2：文件上传乱码**
- 问题：Streamlit file_uploader 上传后存到 tempfile（随机名如 tmpXXX.txt），导入时用的是随机名而非原始文件名
- 解决方案：改为保存到固定 temp 目录下以原始文件名存储，导入后再清理
- 放弃方案：修改 storage.copy_file 传 display_name——增加了不必要的复杂度

### 解决方案与取舍

**产品取舍**：
- 删除 tracker 而非优化：面试追踪的统计看板在求职决策场景有价值，但作为 AI Agent 面试助手的定位，与「准备+模拟」两条核心链路关联太弱，保留会稀释产品焦点
- Obsidian 全链路打通而非单点集成：不只是「能从 Vault 导入」，而是搜索/准备/面试三处都自动感知 Vault 内容

### 产品复盘思考

1. **Agent 产品需要做减法**：v1.0 想把面试全流程都覆盖，但实际使用中 tracker 打开率接近零。v1.1 删除后产品更聚焦，后续迭代也应优先强化核心链路而非扩展覆盖面
2. **外部数据源应该是隐形的**：Obsidian 集成不应该让用户记住「先用 obsidian search 再用 material import」，而应该让用户在 material search 时自动看到 Vault 里的结果。v1.1 做到了这一点
3. **Mock 的简历选择是高频操作**：每次模拟面试都需要指定「我是谁」和「面哪家」，下拉选择比手动输入精准且高效

### 本次使用 AI 技术
- **LLM 选型**：DeepSeek-Chat，温度按场景分层（结构化提取 0.3 / 面试官角色 0.7），支持 JSON Mode
- **Prompt 工程**：4 套系统提示词，分别优化结构化提取、引导式问答、角色扮演、多维度评估场景
- **意图识别**：正则规则引擎，覆盖 16 种中文自然语言意图，支持显式命令和模糊表达两种路由
- **架构设计**：Skill 模块化 + 调度器模式，LLM Provider 抽象层支持多模型切换

### 开发落地难点

**难点 1：模拟面试中追问逻辑的设计**
- 问题：面试官需要在「追问深挖」和「出下一题」之间做判断，过于激进则体验差，过于宽松则评估失真
- 解决方案：不在代码层做规则判断，将追问权完全交给 LLM，通过 Prompt 约束「追问 1-2 轮」的自然节奏，让 LLM 根据回答质量自主决定
- 放弃方案：硬编码追问规则（如关键词匹配回答长度）——过于机械、无法适配不同题型

**难点 2：Rich 终端输出与 Web 渲染的兼容**
- 问题：CLI 用 Rich 库（Panel/Table/Tree）渲染输出，但这些对象无法直接在 Streamlit 展示
- 解决方案：不重构已有 Skill 代码，做 Console 捕获层——在 Web 请求时将 Rich Console 重定向到 StringIO 缓冲区，提取纯文本后清洗 box-drawing 字符，再在 Streamlit 中渲染
- 放弃方案：拆分为独立 Web API 服务——过度工程化，当前个人项目量级不需要

**难点 3：API Key 安全与公网部署的矛盾**
- 问题：配置文件中硬编码了 DeepSeek API Key，推送到公开 GitHub 仓库会造成泄露
- 解决方案：实现三层密钥加载优先级（环境变量 → .env 文件 → config.yaml），推送前移除硬编码 key，Streamlit Cloud 通过 Secrets 管理注入
- 放弃方案：自建后端代理转发——增加延迟和维护成本

### 解决方案与取舍

**技术选型取舍**：
- 选择 DeepSeek 而非 Claude/GPT-4：成本优势显著（约为 1/20），中文 B 端场景表现足够；但牺牲了部分复杂推理能力和工具调用原生支持
- 选择 Streamlit 而非 Next.js：Python 技术栈统一，0 前端代码实现 Web UI；但牺牲了自定义 UI 的灵活性和移动端适配
- 选择 JSON 文件存储而非数据库：零依赖、零配置，个人量级完全够用；但牺牲了并发能力和查询灵活性

**产品设计取舍**：
- Prep 模块「引导而非代答」原则：给框架、要点、素材提示，但不直接写完整答案——让候选人真正学会回答而非背诵
- Mock 硬限制 10 题：保障单次面试效率和成本控制，同时覆盖足够题型检验能力
- 评估报告包含「示范回答」：最差维度给一个高质量示范，平衡批判性和建设性

### 产品复盘思考

1. **Agent 产品的核心壁垒不在模型，在领域知识**：B 端 PM 面试的题型设计、追问节奏、评估维度都依赖对岗位的深度理解，通用面试工具难以覆盖。后续的迭代重点应是题库积累和评估精细度，而非频繁切换模型。

2. **双端策略是正确的**：CLI 适合高频快速操作（查看记录、快速搜索），Web 适合沉浸式体验（模拟面试聊天、画像浏览）。两种形态各有使用场景，不应互相替代。

3. **素材质量决定 Agent 上限**：画像准确性、准备材料针对性、模拟面试个性化程度，全部依赖素材的完整度。下一步应强化素材引导——告诉用户「缺少哪类素材会导致什么能力短板」。

4. **面试追踪是容易忽视但高价值模块**：投递 → 面试 → 结果的全链路数据是个人求职决策的基础，这个模块后续应加入趋势分析和面试策略建议。
